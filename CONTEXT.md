# AICR — Domain Context

## Glossary

### Job
A user's request to clip a YouTube video. Created when a user submits a URL. Has a lifecycle: `pending → processing → ready | error`. A Job holds one `providerProjectId` — the opaque identifier the active provider assigns to the processing task. Jobs are owned by a user (nullable for anonymous submissions).

### Clip
A short video segment extracted from a Job's source video by the AI provider. Clips are generated by the provider and stored locally as metadata only (title, virality score, duration, start/end times). Clip download URLs are ephemeral and fetched live from the provider on demand — never stored.

### Provider
An external AI video clipping service (e.g. Reap). The system is designed to support multiple providers. Each provider implements the `ClipProvider` interface. Only one provider is active at a time, selected at boot. Provider-specific concepts (project IDs, folder IDs, token formats) never leak into domain tables or API responses.

### ClipProvider (interface)
The contract every AI video clipper adapter must implement:
- `createProject(sourceUrl)` → opaque `providerProjectId`
- `getProjectStatus(providerProjectId)` → `'processing' | 'completed' | 'failed'`
- `getClips(providerProjectId)` → `ProviderClip[]` (called once at completion, results persisted)
- `getClipUrls(providerProjectId)` → `Map<clipId, url>` (called on demand, never persisted)

### ProviderClip
The metadata a provider returns at completion, safe to persist: `{ providerClipId, title, viralityScore, duration, startTime, endTime }`. No URLs.

### Default Clip Config
The baseline parameters used for every job submitted via the main flow (not experiments). Defined as a `const` in the Reap adapter — not in environment variables. Current defaults: 30s clips, portrait 9:16, 1080p, captions on, emojis off, results trimmed to 3 clips.

### Experiment
An owner-only A/B testing run: one source video submitted with multiple `Configuration` variations. Each variation becomes a Job. Results (virality scores per clip) are compared across configurations.

### Configuration
The provider-agnostic parameter set users can vary in experiments:
- `clipDuration`: `30 | 60 | 90` (seconds)
- `orientation`: `'portrait' | 'landscape' | 'square'`
- `captions`: `boolean`
- `emojis`: `boolean`

Fields deliberately excluded: `remove_silences` (Klap-specific, no Reap equivalent), `max_clip_count` (not a real API parameter on any current provider — clip count is controlled internally by the adapter).

### Poller
A Redis-backed background loop that monitors active Jobs by calling the provider's `getProjectStatus`. Runs every 30 seconds. When a job reaches `completed`, the poller calls `getClips`, persists the metadata, and removes the job from the queue. The poller is the fallback for missed webhook events.

### Webhook
A push notification from the provider (Reap) when a project reaches a final state. The webhook handler verifies the reported status by re-fetching from the provider before acting. On `completed`, it calls `getClips`, persists metadata, and removes the job from the poller queue. On `failed`, it marks the job as errored. The poller continues running as a safety net.

## Invariants

- Clip URLs are never stored in the database.
- Provider-specific identifiers (`klapTaskId`, `klapFolderId`, etc.) do not appear in domain tables or API responses.
- The `Configuration` schema only contains fields every current provider can honour. Silent no-ops are not permitted.
- The Reap adapter trims clip results to 3 after fetching — this is adapter behaviour, not a domain rule.
- Projects on Reap expire after 60 days (Creator) or 120 days (Studio). After expiry, `getClipUrls` will fail gracefully — the API must surface this to the caller, not silently return empty.
