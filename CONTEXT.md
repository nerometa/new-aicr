# AICR — Domain Context

## Glossary

### Job
A user's request to clip a YouTube video. Created when a user submits a URL and selects a provider. Has a lifecycle: `pending → processing → ready | error`. A Job holds one `providerProjectId` — the opaque identifier the chosen provider assigns to the processing task — and a `provider` field recording which adapter owns it for the lifetime of the job. Jobs are owned by a user (nullable for anonymous submissions).

### Clip
A short video segment extracted from a Job's source video by the AI provider. Clips are generated by the provider and stored locally as metadata (title, virality score, duration, start/end times). Clip download URLs are ephemeral for Reap (fetched live from the provider on demand, never stored) and stored at completion for Reka (populated during `getClips`, served from the `clip_url` column). All virality scores are normalized to 0–100 regardless of provider's native scale.

### Provider
An external AI video clipping service (e.g. Reap, Reka Vision). Users select a provider per job at submission time. Each provider implements the `ClipProvider` interface. Multiple providers can be active simultaneously — the registry is built at boot from configured API keys. Provider-specific concepts (project IDs, score scales, token formats) never leak into domain tables or API responses.

### ClipProvider (interface)
The contract every AI video clipper adapter must implement:
- `createProject(sourceUrl, config?)` → opaque `providerProjectId`
- `getProjectStatus(providerProjectId)` → `'processing' | 'completed' | 'failed'`
- `getClips(providerProjectId)` → `ProviderClip[]` (called once at completion, results persisted; `clipUrl` populated for providers that return URLs at completion)
- `getClipUrls(providerProjectId)` → `Map<clipId, url>` (called on demand; Reap fetches live, Reka re-fetches `GET /v1/clips/{id}` and parses output)

### ProviderClip
The metadata a provider returns at completion, safe to persist: `{ providerClipId, title, viralityScore, viralityScoreExplanation, duration, startTime, endTime, clipUrl? }`. `viralityScore` is always normalized to 0–100 by the adapter (Reap raw × 10, Reka `ai_score` as-is). `clipUrl` is populated only for providers that return stable URLs at completion (currently Reka); Reap leaves it null.

### Default Clip Config
The baseline parameters used for every job submitted via the main flow (not experiments). Defined as a `const` in each provider's adapter — not in environment variables. Reap defaults: 30s clips, portrait 9:16, 1080p, captions on, emojis off. Reka defaults: `moments` template, 3 generations, max 90s, portrait 9:16, subtitles on, 720p. All clips are returned sorted by virality score descending.

### Experiment
An owner-only A/B testing run: one source video submitted with a chosen provider and multiple `Configuration` variations. Each variation becomes a Job under the same provider. Results (virality scores per clip) are compared across configurations within the experiment. Provider is fixed at experiment level — all variants use the same provider, keeping scores internally comparable.

### Configuration
The provider-agnostic parameter set users can vary in experiments:
- `clipDuration`: `30 | 60 | 90` (seconds)
- `orientation`: `'portrait' | 'landscape' | 'square'`
- `captions`: `boolean`

Fields deliberately excluded: `emojis` (Reap-only internal config, no equivalent on Reka — lives in Reap's `DEFAULT_CLIP_CONFIG` only), `remove_silences` (Klap-specific), `max_clip_count` (not a real API parameter on any current provider).


### Poller
A Redis-backed background loop that monitors active Jobs by calling the correct provider's `getProjectStatus` — resolved per-job from `job.provider` via the provider registry. Runs every 30 seconds. When a job reaches `completed`, the poller calls `getClips`, persists the metadata (including `clipUrl` when present), and removes the job from the queue. The poller is the sole completion mechanism for Reka (no webhook support) and the fallback for missed Reap webhook events.

### Webhook
A push notification from Reap when a project reaches a final state. Reap-only — Reka has no webhook support. The webhook handler verifies the reported status by re-fetching from the Reap provider before acting. On `completed`, it calls `getClips`, persists metadata, and removes the job from the poller queue. On `failed`, it marks the job as errored. The poller continues running as a safety net for both providers.

## Invariants

- Reap clip URLs are never stored in the database — they are ephemeral and fetched live via `getClipUrls`. Reka clip URLs are stored at completion in `clips.clip_url` and served directly; `getClipUrls` for Reka re-fetches `GET /v1/clips/{id}` on demand.
- All virality scores stored in `clips.virality_score` are normalized to 0–100. Reap raw scores (0–10) are multiplied by 10 in the adapter. Reka `ai_score` (0–100) is stored as-is.
- Provider-specific identifiers and score scales do not appear in domain tables or API responses.
- The `Configuration` schema only contains fields every current provider can honour. Silent no-ops are not permitted. `emojis` is Reap-internal only.
- The `provider` field on both `jobs` and `experiments` is immutable after creation — a job's provider never changes.
- Projects on Reap expire after 60 days (Creator) or 120 days (Studio). After expiry, `getClipUrls` will fail gracefully — the API must surface this to the caller, not silently return empty.
- `REKA_API_KEY` and `REAP_API_KEY` are both required at boot — this is a hosted service with both providers always available.
